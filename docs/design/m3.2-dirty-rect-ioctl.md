# M3.2 Design — `/dev/fb0` dirty-rect ioctl

**Status:** **IMPLEMENTED 2026-04-30.** virt 181/181, virt-ramfb-fallback 181/181, raspi3b 151/151.
**Commits:** 7a89648 (Commit 1 ioctl plumbing), 96bee3c (Commit 2 fbdev handler), 71fb851 (Commit 3 virtio-gpu coalesced state), 41040b7 (Commit 4 fallback counter), 7dc056b (Commit 5 compositor), this one (Commit 6).
**Scope:** PRD Phase 2 third sub-milestone; replaces M3.1's full-frame TRANSFER+FLUSH every tick with userspace-driven per-rect updates.
**Process:** built from `/octo:embrace` discover/define/debate (codex; gemini failed quota).

## Why

M3.1 made virtio-gpu the active framebuffer provider on the QEMU `virt` arm64 platform, but the deferred-flush pump issued a full-frame TRANSFER_TO_HOST_2D + RESOURCE_FLUSH on every tick at ~50 Hz — ~150 MiB/s of QEMU memcpy, even when the compositor wrote no new pixels. The compositor already does dirty-rect tracking internally (`user/apps/desktop.c:present_dirty_rect`); M3.2 plumbs that tracking through to the kernel via a /dev/fb0 ioctl so the kernel only TRANSFERs what changed.

## Locked design

| Aspect | Choice | Rationale |
|---|---|---|
| ioctl dispatch | `chardev_ops_t.ioctl` member; `syscall_ioctl` falls through to it for FD_TYPE_CHARDEV | Driver-owned dispatch; future virtio-codec/sound get the hook for free |
| Request number | `DRUNIX_FBIO_FLUSH_RECT = 0xD801`; range 0xD800-0xD8FF reserved for fbdev | Drunix-internal ABI; no Linux _IOC compatibility needed |
| Wire format | Pointer to `drunix_rect_t` (one rect per call) | Compositor already produces a single coalesced dirty rect |
| Validation | Kernel rejects NULL, negative coords, zero/negative w/h, OOB | Kernel doesn't trust userspace; -1 with no state change |
| Pump integration | Rect-driven; transitional ~3 Hz fallback for clients without ioctl | M3.1-built compositors still render |
| Kernel queue | Single coalesced rect (union) | Simpler than ring buffer; degenerate case = full-frame is acceptable |
| Pump signature | `arm64_virt_virtio_gpu_pump_flush(void)` unchanged at call sites | Existing `arm64_sync_handler` and `arch_idle_wait` callers don't learn rect semantics |
| Cache clean | Row-span: `g_scanout + rect.y * pitch` for `rect.h * pitch` bytes | Sub-rect not contiguous; row-span over-cleans horizontally but stays correct |
| Fallback trigger | `g_no_ioctl_ticks` counter; threshold 17 (≈340 ms at 50 Hz request rate) | Reset on every publish_dirty_rect; synthesizes full-frame after threshold |
| Compositor change | `present_dirty_rect()` calls `sys_ioctl(g_fb_fd, DRUNIX_FBIO_FLUSH_RECT, &rect)` | One-line addition; fbfd kept open |

## Race protection (codex M3.2 debate-gate concern)

The pump's "consume the pending rect" runs in process context; `publish_dirty_rect` can run from either process context (fbdev ioctl) OR IRQ context (timer-tick fallback). To prevent the consume from tearing against an IRQ-context publish, the pump masks IRQs (`msr daifset, #2`) for the four-int copy + `valid=0` write, then unmasks before the actual TRANSFER+FLUSH (which runs with IRQs enabled per the M3.1 deferred-pump rule).

## Bandwidth impact (measured intent — not benchmarked)

| Workload | M3.1 | M3.2 |
|---|---|---|
| Idle | ~150 MiB/s | ~0 MiB/s (no flushes); ~9 MiB/s if M3.1 binary triggers fallback |
| Mouse movement | ~150 MiB/s | <1% of full-frame (pointer-old ∪ pointer-new) |
| Window drag | ~150 MiB/s | 10–40% of full-frame (titlebar strip / window-bounds union) |

## Out of scope for M3.2

- Multi-rect ioctl batching (one rect per call; coalesced in kernel).
- Write-back framebuffer mmap with precise cache-line cleaning (`CHARDEV_CACHE_WB_FLUSH` stays unimplemented).
- Linux fbdev `_IOC` compatibility.
- Damage tracking inside the kernel beyond union-coalesce.
- Permanently dropping the compatibility fallback (M3.x).
- Hardware cursor (M3.3+).
