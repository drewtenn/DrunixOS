# M3.3 Design — virtio-gpu hardware cursor

**Status:** **IMPLEMENTED 2026-04-30.** virt 184/184, virt-ramfb-fallback 184/184, raspi3b 151/151.
**Commits:** b031020 (Commit 1 shared header + ioctl), b9dd806 (Commit 2 driver-side cursor), 5905b9d (Commit 3 fbdev hook + ioctl), 19b4660 (Commit 4 compositor), this one (Commit 5).
**Scope:** PRD Phase 2 fourth sub-milestone; replaces the compositor's software-drawn cursor with a virtio-gpu cursor plane that the host overlays on the scanout.
**Process:** built from `/octo:embrace` discover/define/debate (codex).

## Why

After M3.0 → M3.1 → M3.2, virtio-gpu owns `/dev/fb0` and uses userspace dirty rects. The compositor (`user/apps/desktop.c`) still composites the mouse cursor sprite into the framebuffer on every pointer move — that's a 13×20 pixel write + a dirty-rect ioctl per move event, and the cursor "follows" the compositor's redraw cadence rather than the host's actual mouse position.

M3.3 offloads the cursor to virtio-gpu's hardware cursor plane: the kernel uploads the sprite once, then the compositor issues a tiny ioctl per pointer move and the host overlays the cursor independently of the scanout. Result: zero compositor framebuffer traffic for pointer-only updates, and the cursor follows the host's mouse with no compositor latency.

## Locked design

| Aspect | Choice | Reason |
|---|---|---|
| Cursor sprite size | 64×64 BGRA; Drunix's 13×20 sprite at top-left, rest transparent | Matches QEMU expectation; 16 KiB; userspace doesn't choose hw-facing detail |
| Sprite backing | `virt_dma_alloc()`, 4 pages | Same DMA model as scanout; isolated from /dev/fb0 |
| Cursor request page | DEDICATED 1-page DMA buffer (`g_cursor_req`) | **Codex M3.3 debate-gate finding**: virtqueue descriptors don't inline payload; reusing controlq's `g_req` for cursorq commands would race with the controlq pump |
| Position update path | Userspace ioctl `DRUNIX_FBIO_MOVE_CURSOR = 0xD802` | Mirrors M3.2's flush ioctl; compositor already integrates relative-mouse motion |
| Wire format | New `drunix_point_t {int x, y}` | Don't overload `drunix_rect_t` with ignored w/h |
| Compositor change | Runtime probe at startup; if move-cursor ioctl returns 0, set `g_hw_cursor_active = 1`, stop drawing software cursor; otherwise keep M2.5b software cursor | Preserves ramfb fallback and virtio-gpu-failure path; one-line opt-in |
| Init sequencing | `virtio_gpu_cursor_init()` after `fbdev_init` succeeds | Cursor sprite upload via controlq, then `UPDATE_CURSOR` via cursorq with screen-center initial position |
| Failure mode | Fail closed: any sprite-upload failure leaves `g_cursor_ready=0` and `fbdev_set_move_cursor` is NOT called → ioctl returns -1 → compositor stays on software cursor | Visible-but-unmovable hardware cursor + software cursor would be worse than clean fallback |
| Cache discipline | Same `arm64_dma_cache_clean` before TRANSFER_TO_HOST_2D as the scanout path | Consistent; mostly academic on Normal-NC |
| Resource ID | `VIRTIO_GPU_CURSOR_RESOURCE_ID = 2u` (scanout uses 1u) | One static cursor for M3.3; userspace sprite upload deferred |

## Race protection

`g_cursor_req` is dedicated to cursorq — no overlap with controlq's `g_req`. cursorq commands are fire-and-forget; the device reads the buffer as it processes the descriptor. Rapid back-to-back submits race on the buffer values, but since all M3.3 traffic post-init is `MOVE_CURSOR` with the same struct shape and only x/y differ, the device sees consistent commands. `virtio_gpu_cursorq_drain_used()` reclaims used-ring descriptor chains at the head of every submit so the queue doesn't fill.

## Performance impact

| Workload | M3.2 (software cursor) | M3.3 (hardware cursor) |
|---|---|---|
| Pointer-only update (mouse drift) | 13×20 area redraw + dirty-rect flush + cursor area in flush | One DRUNIX_FBIO_MOVE_CURSOR ioctl + cursorq submit, no framebuffer touch |
| Window drag | Full window-bounds union flush + cursor sprite drawn | Same minus cursor sprite; cursor follows independently |
| Idle | No flushes | No flushes (cursor still moves freely) |

## Out of scope for M3.3

- Cursor styles (M3.x; would need a sprite-upload ioctl).
- Multi-cursor support.
- Animated cursors.
- Userspace sprite upload from the compositor.
- Hot-spot variations beyond the static (0, 0) hotspot.
- Cursor visibility toggling (always visible after init).
