# M3.1 Design — virtio-gpu owns `/dev/fb0` on QEMU `virt`

**Status:** design, not yet implemented.
**Scope:** PRD Phase 2 second sub-milestone, building on M3.0 (virtio-gpu MMIO discovery + kernel-owned scanout).
**PRD reference:** `docs/superpowers/specs/2026-04-29-gpu-h264-mvp.md` §6.3 / §7 Phase 2.
**Process:** built from `/octo:embrace` discover/define/debate output (codex + gemini).

## Why

Phase 2's PRD exit criterion is: "Drunix's existing compositor draws on screen via virtio-gpu under QEMU; arm64 `virt` no longer needs a GRUB-handoff fbdev." M3.0 brought up virtio-gpu and proved the protocol works (32x32 BGRA test pattern flushed). M3.1 actually retires ramfb on `virt` by giving virtio-gpu ownership of `/dev/fb0`.

After M3.1, the existing compositor (`user/apps/desktop.c`) writes to `/dev/fb0` mmap as it does today; the kernel transparently forwards those writes to the host display via virtio-gpu's `TRANSFER_TO_HOST_2D` + `RESOURCE_FLUSH`. Userspace doesn't change. ramfb stays as the fallback when virtio-gpu init fails or when `-device virtio-gpu-device` is absent.

## What's already there (post-M3.0, on `main`)

- `kernel/arch/arm64/platform/virt/virtio_gpu.{c,h}` — driver with feature-negotiation, controlq + cursorq queue setup, six-command 2D bring-up sequence, polled completion, _Static_asserts on protocol struct sizes, idempotent init, strict response validation (type, flags, fence_id, ctx_id, completed_len), bounded poll timeouts. Currently uses a 32x32 BGRA buffer from the 24-page DMA pool.
- `kernel/arch/arm64/platform/virt/platform_mm.c:51` — `VIRT_RAMFB_BYTES = 8 MiB` framebuffer reservation at top of RAM, classified `PLATFORM_MM_FRAMEBUFFER`, mapped Normal-NC.
- `kernel/arch/arm64/platform/virt/ramfb.c` — calls `fbdev_init()` to publish `/dev/fb0` (M2.5a).
- `kernel/drivers/fbdev.c` — `fbdev_init(const framebuffer_info_t *fb)` registers `/dev/fb0` and `/dev/fb0info`. NOT idempotent.
- `kernel/arch/arm64/start_kernel.c` — init order: `arm64_virt_ramfb_init()` first, then `arm64_virt_virtio_gpu_init()`.

## Locked design choices

These come from the discover synthesis (codex + gemini) and the debate-gate adjustments. Each choice has a one-line "why".

### 1. Reservation strategy: reuse the existing 8 MiB carve-out

virtio-gpu uses `platform_ram_layout()->framebuffer_base/size` as its scanout resource backing. Same physical region ramfb uses today; the 8 MiB reservation already excludes itself from PMM and kheap.

**Why:** zero RAM overhead vs. carving a parallel reservation; the reservation is already physically contiguous, identity-mapped, and tagged Normal-NC for the userspace mmap contract.

### 2. `RESOURCE_ATTACH_BACKING`: single contiguous mem_entry

The reservation is one contiguous physical region. virtio-gpu's `nr_entries = 1` with `addr = framebuffer_base`, `length = pitch * height`.

**Why:** SG list adds complexity for no benefit when the backing is contiguous. Pass the physical address (which equals the kernel-virtual identity-mapped address on `virt`).

**Gotcha:** `length` is the active scanout size (e.g., 1280*800*4 = ~4 MiB), NOT the full 8 MiB reservation, unless future work grows the resource (back buffer, etc.).

### 3. Cache attribute: Normal-NC

The scanout buffer is mapped Normal-NC (existing `PLATFORM_MM_FRAMEBUFFER` policy). User-space mmap of `/dev/fb0` already uses `CHARDEV_CACHE_NC`; preserving this means zero userspace impact.

**Why:** keeps `fbdev_cache_policy()` returning `CHARDEV_CACHE_NC` unchanged; eliminates the per-frame `arm64_dma_cache_clean()` burden TRANSFER_TO_HOST_2D would otherwise require.

**Trade-off:** CPU writes into the framebuffer are uncached, so compositor pixel-blit is slower than it would be with WB. Acceptable for M3.1; revisit in M3.x once the compositor is dirty-rect-correct enough to amortize cache cleans.

### 4. /dev/fb0 swap-over: virtio-gpu first, ramfb skips on success

`start_kernel.c` calls `arm64_virt_virtio_gpu_init()` BEFORE `arm64_virt_ramfb_init()`. ramfb checks `chardev_get("fb0")` at the top and skips if already registered.

**Why:** first successful provider wins. `fbdev_init` stays non-idempotent (single source of truth for `/dev/fb0`); ramfb's existing fallback semantics survive.

**Gotcha:** the skip check lives in `ramfb.c`, not `fbdev.c`. ramfb knows it's the fallback; fbdev_init refuses re-registration cleanly via existing `chardev_register` semantics.

### 5. Failure modes

- **Init failure** (feature negotiation rejected, queue setup fails, six-command sequence fails): virtio-gpu's existing M3.0 cleanup-path code resets the device, frees buffers, returns -1. start_kernel.c falls through to ramfb_init, which publishes `/dev/fb0` as before.
- **Runtime TRANSFER/FLUSH failure**: log (rate-limited; e.g., 1 line per 60 attempts) and continue. **Never swap providers mid-flight** — the compositor's mmap'd region's underlying physical pages must stay constant; userspace has no error return path on raw mmap writes; turning mmap writes into "display dead" failures is a misleading contract. The visible behavior under permanent failure is a frozen/stale display, which is honest.
- **Permanent-disable threshold**: track `g_flush_failures`. After N=60 consecutive failures (~1s of dead display at 60 Hz target), emit a one-shot "virtio-gpu: display permanently disabled" log and stop attempting flushes. Avoids log spam.

### 6. Flush trigger: timer tick sets a flag, deferred non-IRQ pump runs the actual command

**This was the debate-gate's correction.** Earlier proposal had `arm64_timer_tick()` directly calling a polled `virtio_gpu_tick()` that drove the controlq. Codex's M3.1 review flagged this as the real design flaw:

> IRQ polling is the real flaw. A bounded busy wait in timer IRQ context can still create ugly latency, hide scheduler problems, and make timeout behavior harder to reason about.

Corrected design:

- `void virtio_gpu_request_flush(void)` — IRQ-safe; sets `g_flush_needed = 1`. No submission, no polling.
- `void virtio_gpu_pump_flush(void)` — runs from non-IRQ context; if `g_flush_needed` is set, clears it and runs TRANSFER_TO_HOST_2D + RESOURCE_FLUSH for the full scanout.
- `arm64_timer_tick` calls `virtio_gpu_request_flush()` (cheap flag set).
- The pump is called from one of:
  - **(a)** the scheduler/idle path AFTER IRQ exit (process context)
  - **(b)** the syscall return path (process context, runs after every syscall)
  - **(c)** an idle-loop hook
  - Recommendation: **(b)**. Compositor + shell make syscalls regularly; per-syscall overhead is dominated by the syscall itself; the polled-controlq path runs in process context where its bounded busy-wait is semantically OK.
- Cadence: an accumulator skips ticks when SCHED_HZ > target frame rate. At SCHED_HZ=100, every-other-tick request yields 50 Hz.

**Why:** keeps the polled controlq path out of IRQ context; preserves compositor latency under ~20–40 ms; doesn't require a kernel-thread infrastructure that Drunix doesn't have today.

### 7. Dirty-rect: full-frame for M3.1, ioctl-driven for M3.2

Every pump call issues `TRANSFER_TO_HOST_2D` + `RESOURCE_FLUSH` for the entire scanout rectangle. A 4 MiB transfer at 60 Hz = ~240 MiB/s of QEMU memcpy, well within QEMU performance envelopes for local development.

M3.2 will add a `/dev/fb0` ioctl carrying user-space dirty-rect metadata. The compositor already does dirty-rect tracking internally (`shared/desktop_window.h` rectangle helpers); M3.2 just plumbs that through.

**Why:** M3.1 is the provider swap, not presentation optimization.

### 8. KTEST changes

**Existing tests to rename/retarget** (semantic, not just rename):

- `test_arm64_ramfb_layout_reserves_8mib` → `test_arm64_framebuffer_reservation_is_8mib`
- `test_arm64_ramfb_classifier_returns_framebuffer` → `test_arm64_framebuffer_reservation_classifier`
- `test_arm64_ramfb_kernel_alias_is_normal_nc` → `test_arm64_framebuffer_kernel_alias_is_normal_nc`
- `test_arm64_fbdev_geometry_matches_ramfb_config` → `test_arm64_fbdev_geometry_matches_active_provider_config` (geometry now comes from virtio-gpu's display-info report on the gpu path)
- `test_arm64_virtio_gpu_dma_pages_held_within_budget`: assertion changes from `held == 7u` to `held == 6u` (scanout no longer in DMA pool — moved to the framebuffer reservation).

**New tests:**

- `test_arm64_virtio_gpu_owns_fb0_post_init` — after `arm64_virt_virtio_gpu_init()` succeeds, `chardev_get("fb0")` returns non-NULL AND the published `framebuffer_info_t.phys_address` equals `platform_ram_layout()->framebuffer_base`.
- `test_arm64_virtio_gpu_pump_flush_idempotent_when_no_request` — calling `virtio_gpu_pump_flush()` without a prior `virtio_gpu_request_flush()` is a no-op (no submission, no log).
- `test_arm64_virtio_gpu_pump_flush_clears_request_flag` — after pump runs, `g_flush_needed == 0`.

**Fallback-path harness** (cannot fault-inject from within a single kernel binary): add `tools/test_arm64_ktest_ramfb_fallback.py` that runs QEMU WITHOUT `-device virtio-gpu-device` but WITH `-device ramfb` and asserts the boot log shows ramfb publishing /dev/fb0.

## Implementation sequence (revised after delivery review)

Codex's M3.1 delivery review (`/tmp/codex-m3.1-deliver.md`) flagged that the original 8-step sequence was not buildable green step-by-step. Steps now grouped into **commits** that each leave the test suite green. The dependency rules:

- The init-order swap (`start_kernel.c`) and the ramfb skip check (`ramfb.c`) MUST land before virtio-gpu starts calling `fbdev_init()`, otherwise either ramfb wins and virtio-gpu's call fails, or both try and the second logs a registration failure.
- The KTEST renames/edits MUST land in the same commit as the geometry/page-budget changes they cover, because existing tests hard-code ramfb's `1024x768 / pitch 4096` and `virtio_gpu_dma_pages_held == 7`.

### Commit 1 — Conceptual rename (no behavior change)

- Rename "ramfb reservation" comments in `platform_mm.c` to "framebuffer reservation".
- Rename existing KTESTs that name ramfb without testing ramfb-specific behavior:
  - `test_arm64_ramfb_layout_reserves_8mib` → `test_arm64_framebuffer_reservation_is_8mib`
  - `test_arm64_ramfb_classifier_returns_framebuffer` → `test_arm64_framebuffer_reservation_classifier`
  - `test_arm64_ramfb_kernel_alias_is_normal_nc` → `test_arm64_framebuffer_kernel_alias_is_normal_nc`
- Build + KTEST green: virt 178/178, raspi3b 151/151 unchanged.

### Commit 2 — Init reorder + ramfb skip check (no /dev/fb0 contract change yet)

- `start_kernel.c`: move `arm64_virt_virtio_gpu_init()` call BEFORE `arm64_virt_ramfb_init()`.
- `ramfb.c`: add `if (chardev_get("fb0")) { platform_uart_puts(...); return 0; }` at the top of `arm64_virt_ramfb_init()`.
- At this point virtio-gpu still uses the small DMA-pool scanout (M3.0's 32×32) and DOES NOT call `fbdev_init()`, so ramfb still wins and publishes `/dev/fb0`. The skip check fires only after Commit 3.
- Build + KTEST green.

### Commit 3 — virtio-gpu owns the framebuffer reservation + KTEST adjustments

This is the load-bearing commit. Lands together because the geometry/page-budget invariants change atomically:

- `virtio_gpu.c`:
  - Replace the M3.0 32×32 DMA-pool scanout with `platform_ram_layout()->framebuffer_base` as the resource backing.
  - Read width/height from GET_DISPLAY_INFO; pass them to `RESOURCE_CREATE_2D` and `SET_SCANOUT`.
  - Build a `framebuffer_info_t` and call `fbdev_init(&fb)` after `RESOURCE_FLUSH` succeeds.
  - Adjust `g_dma_pages_held` accounting: 2 controlq + 2 cursorq + 1 req + 1 resp = 6 pages (was 7).
- `test_arch_arm64.c`:
  - Update `test_arm64_virtio_gpu_dma_pages_held_within_budget`: assert `held == 6u` (was 7).
  - Add `test_arm64_virtio_gpu_owns_fb0_post_init`: `chardev_get("fb0") != NULL` AND the published framebuffer's phys equals `platform_ram_layout()->framebuffer_base`.
  - Rename `test_arm64_fbdev_geometry_matches_ramfb_config` → `test_arm64_fbdev_geometry_matches_active_provider_config` and update assertions to expect virtio-gpu's reported geometry on the gpu path.
- After this commit, virtio-gpu owns `/dev/fb0` on the gpu-enabled path; ramfb fallback still works when virtio-gpu init fails (the existing M3.0 cleanup path falls through to ramfb).
- Build + KTEST green: virt count grows by ~1-2 (one new test, some renames don't change count).

### Commit 4 — Deferred flush mechanism (timer-tick request, syscall-return pump)

- `virtio_gpu.h`: export `virtio_gpu_request_flush()` and `virtio_gpu_pump_flush()`.
- `virtio_gpu.c`: implement both. `g_flush_needed` is `volatile uint32_t`. `g_flush_failures` counter. Permanent-disable threshold at 60 consecutive failures; one-shot log; clear `g_ready` to make `arm64_virt_virtio_gpu_ready()` return 0.
- `start_kernel.c` `arm64_timer_tick`: call `virtio_gpu_request_flush()` with an accumulator that throttles to ~50 Hz (every 2nd tick at SCHED_HZ=100).
- **Pump trigger location** (codex flagged that this hook doesn't exist as named today):
  - Closest existing location: `arm64_sync_handler()` after `syscall_dispatch_from_frame()` and signal check, before returning to `eret` via `exceptions_s.S`. Add the `virtio_gpu_pump_flush()` call there. Confirm it runs in process context with IRQs unmasked (or unmasks them if needed; bounded busy-wait inside the polled controlq is safe in process context).
  - Fallback / additional trigger if syscall-return alone proves too coarse: add a low-rate fallback pump from the idle wait — but only as M3.1.x polish if observed. Don't speculatively add both.
- KTESTs:
  - `test_arm64_virtio_gpu_pump_flush_idempotent_when_no_request` — call pump without prior request; assert no submission, no log.
  - `test_arm64_virtio_gpu_pump_flush_clears_request_flag` — request, pump, assert `g_flush_needed == 0` afterwards.
- Build + KTEST green.

### Commit 5 — ramfb-fallback harness

- `tools/arm64_qemu_harness.py`: factor the QEMU command-line out of `qemu_command(platform)` so the device list is parameterizable.
- New `tools/test_arm64_ktest_ramfb_fallback.py`: invokes the harness with the device list MINUS `-device virtio-gpu-device`; asserts boot log shows `ramfb: ... published as /dev/fb0` AND no `virtio-gpu: ... M3.1 sequence complete` lines.
- New `mk/checks.mk` target wiring so `make check` runs both harnesses.
- Build + KTEST green; both harnesses pass.

### Commit 6 — Documentation refresh

- Update `docs/m3.1-virtio-gpu-fbdev-swap.md` with "Status: implemented" + commit hashes.
- Memory note refresh once branch lands on main.

## Pump-hook details (codex review correction)

**Codex flagged that the syscall-return hook this design references doesn't exist as a named abstraction today.** Concrete plan for Commit 4:

- The closest existing arm64 hook is in `kernel/arch/arm64/exceptions.c` `arm64_sync_handler()`, AFTER `syscall_dispatch_from_frame()` runs the syscall body and AFTER any pending-signal check, BEFORE the return path through `exceptions_s.S` calls `eret`.
- The pump call inserts there. Concrete one-liner: `virtio_gpu_pump_flush();` after the existing post-syscall logic. The function is a no-op when `!g_flush_needed`, so the syscall hot path takes a single load + branch overhead.
- Idle path: `kernel/arch/arm64/timer.c` or `kernel/proc/sched.c` `arch_idle_wait()` — the call goes BEFORE `wfi` so the idle thread services any pending flush before sleeping. This is a small additional insertion point; can ship in Commit 4 alongside the syscall-return hook for completeness, or be deferred to Commit 4.x.
- These hook points are NEW Drunix idioms; document the choice in the commit message so future virtio-codec / virtio-sound drivers can follow the same pattern.

## KTEST harness — fallback variant (codex review correction)

The current `tools/arm64_qemu_harness.py:qemu_command()` hard-codes both `-device ramfb` AND `-device virtio-gpu-device`. The fallback test cannot just add a kernel-side assertion; it needs a QEMU command-line variant.

Concrete refactor:

- Add a `kwarg` to `qemu_command(platform, devices=None)` where `devices` is a list overriding the default device set.
- Default `devices` (gpu path): `["virtio-blk-device", "ramfb", "virtio-keyboard-device", "virtio-mouse-device", "virtio-gpu-device"]` — current behavior.
- `tools/test_arm64_ktest_ramfb_fallback.py`: passes `devices = [..., "ramfb"]` (drop `virtio-gpu-device`).
- Both harnesses share the rest of the boot/wait/SUMMARY logic.

## Acceptance criteria

- `make ARCH=arm64 PLATFORM=virt run` on Apple Silicon shows the **Drunix desktop via virtio-gpu** (compositor pixels appear via TRANSFER+FLUSH, not via ramfb's host-scans-guest-pages).
- `chardev_get("fb0")` after `arm64_start_kernel` on the gpu-enabled path returns the virtio-gpu-backed entry; `framebuffer_info_t.phys_address == platform_ram_layout()->framebuffer_base`.
- Fallback boot (no `-device virtio-gpu-device`) still publishes `/dev/fb0` via ramfb and shows the desktop.
- KTEST (gpu path): pass count grows by ≥ 2 (new ownership + pump tests). No regressions. Existing virt 178/178 → ~181/181.
- KTEST (ramfb fallback path): new harness passes.
- raspi3b 151/151 unchanged.
- The polled controlq wait does NOT execute in IRQ context (verified by code review; a bounded busy wait in `virtio_gpu_pump_flush` is OK because the pump runs in process context).

## Out of scope for M3.1

- WB framebuffer mappings.
- Dirty-rect ioctl (M3.2).
- Page-fault dirty tracking (M3.x).
- Double buffering (compositor stays single-buffered like ramfb today).
- Runtime swap from virtio-gpu back to ramfb if gpu fails after init.
- virtio-gpu IRQ-driven completion (M3.0 polled stays; the deferred pump just runs polled in process context).
- Hardware cursor (M3.3).

## Risks and mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Tearing visible to the user (compositor mid-frame, pump fires, host sees half-frame) | High at 50–60 Hz; equal to ramfb's existing tearing | Acceptable for M3.1. Atomic-present is a separate work item. |
| Pump latency variance because syscall-return is the only trigger and a quiet system makes few syscalls | Medium | Add a low-rate fallback pump from the idle loop (e.g., once every 100 ms) so a fully idle system still presents at some minimum rate. Consider for M3.1 if syscall-return-only feels too coarse during dev. |
| Permanent-disable threshold fires too eagerly (60 timeouts in 1s) when QEMU is paused (e.g., gdb breakpoint) | Low | Document; consider also exposing a "/proc/virtio-gpu/status" or similar for diagnostics. Don't crash; the threshold only stops the spam, doesn't destroy state. |
| Resource size exceeds the reservation (future 1920x1080x32 = 8.3 MiB > 8 MiB reservation) | Low at current target sizes | Bump `VIRT_RAMFB_BYTES` to 16 MiB before raising max display dimensions. |
| Compositor write inside the active TRANSFER's source rect is read torn by the host | Medium | Acceptable; matches ramfb. Atomic-present is M4+ work. |

## What this design doc commits to

- The eight implementation steps above.
- The KTEST renames and additions.
- The deferred-flush mechanism (request flag + non-IRQ pump).
- The reservation reuse (no parallel allocation).
- The Normal-NC cache attribute (no userspace policy change).

## What this design doc explicitly does NOT commit to

- The flush cadence (50 Hz vs 60 Hz) — implementation tunes the accumulator at SCHED_HZ=100.
- The permanent-disable threshold (60 vs other) — implementation picks based on observed tolerable error rates.
- Whether to keep a kernel-side test pattern (M3.0's color quadrants) before fbdev_init runs, or zero the buffer — both are acceptable; pick whatever is least confusing during boot.
- Whether the idle-loop pump trigger ships in Commit 4 alongside the syscall-return hook, or as Commit 4.x — depends on whether syscall-return alone proves coarse enough during dev.
