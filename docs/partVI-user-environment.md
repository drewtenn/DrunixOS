# Part VI — The User Environment

Part V established that programs can run. What it did not establish is that they can do anything useful from a user's perspective. A process that launches, computes silently, and exits is not an interactive system — it is a batch job. For the OS to feel like an operating system in the way users experience one, processes need to respond to keystrokes in a principled way, react to events that arrive while they are in the middle of something else, and exist within a shell that lets the user compose, launch, and manage them.

The gap between "programs run" and "users can work" spans four distinct problems. The first is the terminal: raw keyboard characters are not enough for interactive programs. A text editor needs to know about line boundaries, backspace, and arrow keys. A shell needs to echo what the user types, accumulate a full line before acting on it, and hand the terminal off to the foreground program without losing control of it. The TTY line discipline — the layer of software that sits between the keyboard driver and the programs reading from the terminal — formalises all of this into a configurable set of rules governing how characters flow. The second problem is signals: a running program needs a way to receive asynchronous notifications — that the user pressed Ctrl+C, that a pipe it was writing to has no reader, that a child process has exited — without constantly checking for them. Signals are the kernel mechanism that delivers those notifications at the exact moment the process is about to return to user code. The third problem is the user-space foundation: before any user program can be written as ordinary C, there must be a startup routine that sets up the stack, calls `main`, and issues an exit syscall when `main` returns, plus a library of syscall wrappers so programs do not have to write raw assembly. The fourth is the standard library: programs expect `printf`, `malloc`, `fopen`, and `strcmp` to simply exist. The fifth is the shell itself, which ties all of the above together into the surface the user actually touches.

Chapter 18 builds the TTY line discipline: raw and canonical input modes, the `termios` configuration interface, non-busy blocking reads, and the foreground process group mechanism that determines which process owns the terminal at any moment. Chapter 19 builds the signal subsystem: the pending and blocked signal bitmasks in the process descriptor, the delivery window just before every return to user code, the signal frame that redirects execution to a user handler and returns cleanly, and the stop and continue signals that underpin job control. Chapter 20 builds the user-space runtime: the `crt0` entry point that the linker calls before `main`, the syscall wrapper library, and the `sbrk`-based heap. Chapter 21 builds a minimal C standard library on top of that runtime: string and memory functions, character classification, a sink-based `printf`, `FILE` streams, and POSIX wrappers. Chapter 22 builds the interactive shell: readline with tab completion, command tokenisation, `fork`/`exec`/`wait`, pipe construction, environment management, and the `jobs`, `fg`, and `bg` builtins that let the user manage stopped and background processes.

By the end of Part VI the OS has a face. The user sits at a prompt, runs programs, interrupts them with Ctrl+C, suspends them with Ctrl+Z, and resumes them with `fg`. Programs written in C with ordinary standard library calls compile and run without modification. The system is usable.
