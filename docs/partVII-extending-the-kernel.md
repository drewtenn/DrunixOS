# Part VII — Extending the Kernel

At the end of Part VI the OS is complete in the sense that matters to a user: programs run, the shell is interactive, files persist across reboots. But the kernel is a closed system. Adding new functionality — a driver for a new device, a new filesystem, a new system call — means modifying kernel source code, recompiling the entire kernel, and rebooting the machine. For a developer iterating on kernel components, or for a system that needs to load drivers on demand without restarting, that is an unacceptable constraint. And when something does go wrong — when a program crashes — there is currently no record of what the machine state looked like at the moment of the fault. The process disappears cleanly, and whatever went wrong with it disappears too.

This part addresses both the closed nature of the compiled kernel and the opacity of process crashes. The two problems share a common thread: both require the kernel to understand ELF binaries — the standard executable and object file format — more deeply than just loading their program segments. A loadable module is an ELF relocatable object (a `.o` file, the kind a compiler produces before linking) with unresolved symbol references that the kernel must patch at load time by looking up the target addresses in a table of exported kernel symbols. A core dump is an ELF core file — a binary the kernel *writes*, not reads — whose structure must match what a debugger like GDB expects so it can reconstruct the exact state of the crashed process without any assistance from the running system.

Chapter 23 builds the runtime module loader: the ELF relocatable section parser, the two relocation types the x86 calling convention requires (one for absolute address references and one for relative ones), the kernel symbol export table, the `module_init` call that runs after relocation completes, and the `SYS_MODLOAD` syscall that user programs use to trigger a load. Chapter 24 builds the core dump writer: the ELF core file layout, the register-state note that carries the full CPU register snapshot at the moment of the fault, the process-info note that carries the process name and command line so GDB can identify the binary automatically, and the memory segments that snapshot every user-space page. The process name and argument string are captured at process creation time so they survive whatever the program may have done to its own stack before crashing.

By the end of Part VII the kernel can grow without recompilation, and crashed processes leave behind a complete, analysable record. The gap between the kernel as a static artifact and the kernel as a living, inspectable system is closed.
