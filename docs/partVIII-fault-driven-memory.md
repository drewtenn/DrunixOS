# Part VIII — Fault-Driven Memory

Part II built the first version of memory management: a bitmap-backed physical allocator, a fixed identity map, and enough paging support to launch a user process with its code, data, heap start, and a four-page stack. That was enough to get ring-3 code running, but it also froze the memory story in its simplest form. A larger heap immediately consumed physical memory even if the program never touched it. `fork()` copied every user page up front even when the child was about to `exec()` something else. A write to a shared page could not be expressed as "copy this page now"; the kernel had no place where such a decision could even be made.

The missing piece is the page-fault path. On x86, the CPU does not only use page faults to report bugs. It also uses them to report that translation state is incomplete and ask the kernel to finish the job. A not-present fault can mean "this heap page has never been committed"; a protection fault on a writable mapping can mean "this page is copy-on-write, give this process a private copy"; and a fault just below the user stack can mean "the stack needs one more page". Once the kernel can distinguish those cases, a fault becomes part of normal execution rather than the end of it.

Chapter 25 builds that decision path: how the CPU reports the fault through `CR2` and the page-fault error code, how the kernel turns a missing heap or stack page into a live mapping, and how the old SIGSEGV path remains the fallback for wild pointers and true protection violations. Chapter 26 then uses the same machinery to make `fork()` cheap in the common case by sharing pages until the first write, while the physical memory manager keeps a per-frame reference count so shared pages can be reclaimed safely.

By the end of Part VIII, the memory subsystem stops being a static mapping phase that happens only at process creation time. It becomes a live conversation between the CPU, the page tables, and the kernel's policy code.
