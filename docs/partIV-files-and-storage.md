# Part IV — Files and Storage

The ATA driver from Part III can read and write disk sectors, but a sector number is not a name. To open a program or a document, the user — and every program they run — thinks in terms of paths like `bin/shell` or `docs/notes.txt`, not in terms of sector 1,847 on drive 0. Translating between those two worlds is the job of the filesystem.

A filesystem is fundamentally a map: given a name, find the data. Building that map on top of raw sectors requires agreeing on where the map itself is stored, how it is structured, and how it is kept consistent when files are created, grown, or deleted. The design choices made here have long-lasting consequences. An inode-based layout, where each file's metadata lives in a fixed-size record at a known location, makes lookup fast and makes file sizes independent of where the name appears in the directory. A bitmap to track free blocks keeps allocation O(1). Indirect block pointers let files grow beyond what fits in the inode itself. These are not arbitrary choices — they are the same tradeoffs that Unix filesystems have refined over fifty years.

The second problem this part addresses is abstraction. A kernel that bakes filesystem details into every piece of code that touches storage becomes fragile: adding a second filesystem type, or changing the on-disk format, means touching code everywhere. The Virtual File System layer — the VFS — solves this by placing a thin operations table between callers and the filesystem implementation. Everything above the VFS talks to helpers such as `vfs_open_file`, `vfs_read`, and `vfs_create`; everything below the VFS implements those operations for a specific on-disk format. Swapping or adding filesystem drivers becomes a matter of registering a new operations table, not rewriting the callers.

Chapter 13 builds DUFS — the project's native inode-based filesystem — from the superblock through the inode and block bitmaps, the block pointer structure, directory entries, and path resolution. It then introduces the Linux-compatible ext3 root filesystem that Drunix boots by default, including its ext2-style inode layout and its internal JBD journal. Chapter 14 wraps both filesystems in the VFS layer, defining the operations table that higher-level code uses from this point forward.

By the end of Part IV the disk has structure. Paths resolve to inodes, inodes point to data blocks, and the kernel can create, read, write, and delete named files in a directory hierarchy. The default root image is an ext3 filesystem that host Linux tools can inspect with `e2fsck`, `dumpe2fs`, and `debugfs`; DUFS remains available as the smaller teaching filesystem and as the `/dufs` secondary mount during ext3 boots. The storage layer is ready to serve the process and shell layers that come next.
