# Part V — Running User Programs

Everything built so far runs in the kernel. The interrupt handlers, the memory allocator, the filesystem, the ATA driver — all of it executes at full privilege with unrestricted access to every memory address and every I/O port. That is appropriate for the kernel itself, but it is not acceptable for programs written by users. A bug in a user program must not be able to corrupt the kernel's data structures, overwrite another process's memory, or halt the machine. The CPU's privilege system exists precisely to enforce this boundary, but enforcing it requires the kernel to build the infrastructure to cross it safely.

Running a user program is not a single operation — it is a pipeline of decisions and data structures that must all be in place before the CPU can execute the first instruction of user code. The kernel must parse the ELF binary (the standard executable file format on Linux and Unix systems) from disk, allocate a private page table that maps the program's code and data at the virtual addresses the linker chose, and build a user-mode stack with the program's arguments. It must then configure the CPU's privilege-separation state so that the hardware knows which stack to use when an exception or system call pulls execution back into the kernel. Finally, it issues a return-from-exception instruction with the right register values to drop from kernel privilege to user privilege. Once that program is running, every request it makes — writing output, opening a file, creating a child process — must travel back through a controlled entry point into the kernel and out again.

Chapter 15 builds the full process model: the ELF loader, per-process page tables, the `process_t` descriptor, the round-robin preemptive scheduler driven by the timer tick established in Part I, and `fork` which clones a running process completely. Chapter 16 establishes the system call interface — the kernel entry point, the calling convention, the blocking-state model for calls that must wait for external events, and the complete table of syscalls the kernel exposes. Chapter 17 builds the file descriptor layer on top of the Virtual File System: the per-process open-file table, the unified read and write path that handles both terminal and file I/O, pipes, `dup2`, and seek.

By the end of Part V the kernel is a true operating system. It can load an ELF binary from disk, run it in a private address space at user privilege, preempt it on a timer tick, schedule multiple processes concurrently, and serve their requests through a stable system call interface. The next part will give those processes a way to interact with the user.

The process model, scheduler, and system call interface described in these chapters generalise across architectures; the chapters note where x86 and arm64 differ in the mechanics of privilege transitions and exception entry.
