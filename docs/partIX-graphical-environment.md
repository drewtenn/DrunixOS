# Part IX — Graphical Environment

Up to this point the kernel has been a serious system that talks to its user almost entirely in text. The keyboard from Chapter 10 feeds characters into a ring buffer, the shell from Chapter 22 reads them back, and the VGA text console from Chapter 9 paints whatever the shell prints. None of those layers know anything about pixels, windows, or pointers — and for everything the book has built so far, they have not needed to.

Part IX is where that changes. By the end of the previous part, the memory subsystem is mature enough to support every consumer the desktop will need: page-faulted heap pages back the back buffer's spillover, copy-on-write fork lets the shell launch helper processes cheaply, and on-demand stack growth absorbs the deeper call stacks of the windowed apps. The processes layer from Chapter 15 already gives us pids and process groups. The shell, the file system, and the procfs module (a synthetic filesystem that exposes live kernel state as readable files under `/proc`) all already work. What is missing is a way for the user to *see* and *click on* any of it.

Chapter 27 fills the input half of that story. It adds a PS/2 mouse driver — a second interrupt-driven input device that mirrors the keyboard's architecture but needs new machinery for multi-byte packet reassembly, motion coalescing, and scaled pixel-coordinate tracking. Chapter 28 fills the output half. It builds a compositor on top of either a legacy VGA text buffer or a modern VESA framebuffer, with an off-screen back buffer for tear-free presents, an overlay-sprite mouse cursor, a small window manager, and a shell terminal that lets the existing command-line shell appear as just another desktop app.

By the end of Part IX, the machine looks and feels like an operating system from the user's side of the glass: a desktop with a taskbar, a launcher, mini-apps for browsing the filesystem and live processes, and a windowed terminal hosting the shell.
