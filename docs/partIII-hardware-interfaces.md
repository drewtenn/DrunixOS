# Part III — Hardware Interfaces

With memory under control, the kernel can allocate data structures reliably — but it is still isolated. It cannot observe its own behaviour beyond what fits on the screen, it cannot receive input from the user, and it cannot reach the disk. A kernel that cannot read keystrokes or access persistent storage is not yet an operating system; it is a program that runs once and forgets everything the moment the power goes off.

This part is about giving the kernel senses. The problem has two dimensions. One is mechanical: each piece of hardware speaks its own protocol — the keyboard talks over the PS/2 port (the older round-connector interface for keyboards and mice) using scancodes (numeric codes the keyboard sends to indicate which key was pressed or released), the ATA disk controller (the standard interface that connects hard drives to the motherboard) exposes status and data registers through a pair of I/O port ranges, and both use interrupts to signal that they need attention. The other dimension is architectural: as the number of drivers grows, they must not interfere with each other or with the rest of the kernel. A keyboard handler that reaches into disk driver state, or a disk driver that assumes a particular interrupt vector, becomes a maintenance problem. The solution is a pair of registries — one for block devices like disks, one for character devices like the keyboard — that let drivers advertise a uniform interface without knowing anything about each other.

Chapter 9 establishes the kernel boot log: a structured, tagged logging facility that every subsequent driver uses to report its initialisation status. This is built before the drivers themselves because an uninstrumented driver is nearly impossible to debug. Chapter 10 implements the PS/2 keyboard driver: scancode decoding, modifier key tracking, a ring buffer that the rest of the kernel reads from, and the special handling for Ctrl+C and Ctrl+Z that the terminal layer will later formalise. Chapter 11 implements the ATA disk driver using Programmed I/O — a mode where the CPU directly polls the controller and moves data through I/O ports rather than using a DMA transfer — exposing a simple read/write interface over Logical Block Addressing (LBA), where every sector on disk is identified by a single sequential number rather than a cylinder, head, and sector triplet. Chapter 12 introduces the block and character device registries that abstract over individual drivers, so that the filesystem and terminal layers can talk to "a disk" and "a keyboard" without caring which specific driver is underneath.

By the end of Part III the kernel has eyes and hands. It can log structured messages to the console, accept characters from the keyboard, and read or write any sector on the disk. The building blocks for persistent storage are in place.

The driver model and device registry design described in these chapters is written to be portable; the specific I/O port protocols are x86 hardware details that the architecture-specific driver code handles below the registry boundary.
