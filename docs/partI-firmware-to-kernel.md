# Part I — From Firmware to Kernel

When the machine powers on, it is not running your code. The CPU starts in real mode — a 16-bit addressing mode inherited from the 1970s — and begins executing firmware stored in ROM (read-only memory on the motherboard). That firmware is responsible for initializing the hardware and finding something to boot. What it finds is GRUB, and what GRUB finds is the kernel binary. By the time GRUB hands control over, the CPU is still in a constrained state: 32-bit protected mode is enabled, but there is no interrupt table, no configured segment descriptors beyond what GRUB provided, no way to handle exceptions, and no access to any of the extended CPU features that modern code depends on. The machine is alive but barely governable.

This part is about taking full control of that machine. The problem is not any single missing piece — it is that several pieces must all be in place before any of them can be used safely. The CPU needs a properly configured Global Descriptor Table — a structure that defines the memory regions kernel and user code are allowed to use — before it can enforce privilege levels. It needs an interrupt descriptor table — a lookup table that tells the CPU what code to run when an exception or hardware event fires — before any hardware event can be handled without crashing. It needs the programmable interrupt controller remapped before hardware interrupt requests arrive on the right numbered vectors. And it needs the floating-point and vector instruction extensions enabled before any code that touches those registers can run. None of these steps produces a visible result on their own; together they bring the kernel to a state where everything that follows is possible.

Chapter 1 follows the hand-off from GRUB: what the Multiboot protocol provides, where the kernel binary lands in memory, what framebuffer mode the kernel requests, and what the linker script must guarantee before the first C instruction executes. Chapter 2 builds the Global Descriptor Table and makes the flat 32-bit memory model explicit, establishing the segment selectors that all kernel and user code will use from this point forward. Chapter 3 takes the first output: the kernel writes to the VGA text buffer to confirm it is running, establishes a scrollback shadow buffer, and later presents the shell through a framebuffer-backed desktop when GRUB provides a supported linear framebuffer. Chapter 4 installs the interrupt descriptor table and remaps the 8259 Programmable Interrupt Controller — the chip that routes hardware signals from devices into CPU interrupts — so that device interrupts arrive on vectors that do not collide with CPU exceptions. Chapter 5 adds a dispatch registry on top of the raw interrupt descriptor table so that individual drivers can register handlers without knowing the interrupt vector layout, and wires up the Programmable Interval Timer to produce a periodic tick. Chapter 6 enables the SSE2 floating-point extensions — a set of CPU instructions that operate on wide registers used for math-heavy code — by setting the control register bits that gate their use, then captures a clean floating-point state template that every future process will inherit.

By the end of Part I the machine is fully under kernel control. Exceptions and hardware interrupts are handled. The address space is flat and well-defined. The CPU's extended instruction set is available. The path is clear to build everything else on top.
