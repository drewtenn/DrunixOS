# Part I — From Firmware to Kernel

When a machine powers on, the CPU is not running our code. Firmware runs first — a permanently installed program baked into the hardware whose job is to initialize the machine to a usable baseline and find something to boot. Once it does, it hands control to a bootloader, and the bootloader hands control to the kernel. By the time that hand-off reaches us, some initial machine state is already established, but the kernel is still far from ready. There is no reliable output, no way to handle exceptions or hardware events, and none of the extended CPU features that modern code depends on. The machine is alive, but it is not yet governable.

Part I covers two starting points, and this book treats neither as primary. On an x86 PC, the firmware is the **BIOS** (Basic Input/Output System, software in ROM that initialises the hardware and searches for a bootable device), the bootloader is **GRUB** (GNU GRand Unified Bootloader, the program that loads Linux and other operating systems from disk), and the CPU enters our kernel in 32-bit protected mode (the x86 operating mode that exposes the full 4 GB address space and enforces privilege boundaries between code regions) carrying a Multiboot info pointer — a small data structure GRUB prepared to describe the machine's memory map and display capabilities. On an ARM-based machine — modelled here as the QEMU `raspi3b`, a software emulation of the Raspberry Pi 3 — there is no traditional BIOS or bootloader: QEMU loads the kernel image at a fixed physical address and all four cores begin executing immediately, starting in **EL2** (Exception Level 2, a privileged CPU mode ordinarily reserved for hypervisors). The two paths look nothing alike, yet they arrive at the same problem: the kernel must finish what firmware started and bring the CPU into a state where everything that follows is possible.

This part is about solving that problem. The difficulty is not any single missing piece — it is that several pieces must all be in place before any of them can be used safely. The kernel needs to understand the segmentation model the CPU enforces (or, on AArch64 — the 64-bit execution state of the ARM architecture — confirm that there is none). It needs a first reliable way to write output so it can report what is happening. It needs an exception and interrupt table installed before any hardware event can be handled without crashing. It needs periodic hardware events wired up before a scheduler tick is possible. And it needs the floating-point and vector instruction extensions unlocked before any code that touches those registers can run. None of these steps produces a dramatic result on its own; together they carry the kernel from a bare hand-off to a machine that is fully under our control.

Chapter 1 follows the firmware-to-kernel hand-off: how the bootloader finds and loads our binary, what state it leaves the CPU in, and what the kernel must guarantee before its first instruction executes. Chapter 2 examines the x86 segmentation model — the set of rules that describe how the CPU partitions memory and enforces privilege — and establishes the flat descriptor layout the rest of the kernel will depend on. AArch64 has no segmentation; this chapter notes where the two architectures diverge. Chapter 3 takes the first reliable output: the kernel confirms it is running by writing to a display, giving us a narrator for every bootstrap step that follows. Chapter 4 installs the exception and interrupt table — the structure the CPU consults whenever a fault occurs or a hardware event fires — and configures the interrupt controller so that hardware signals arrive on the right numbered vectors. Chapter 5 builds a dispatch layer on top of that table so individual drivers can register handlers without knowing the interrupt vector layout, and wires up a hardware timer to produce the periodic tick the scheduler will later depend on. Chapter 6 enables the floating-point and vector unit — the part of the CPU that handles extended numeric instructions — by setting the control bits that gate its use, then captures a clean state template that every future process will inherit.

By the end of Part I the kernel is in full control of the CPU on either supported architecture. Exceptions and hardware interrupts are handled. The address space model is understood and configured. The CPU's extended instruction set is available. The path is clear to build everything else on top.
