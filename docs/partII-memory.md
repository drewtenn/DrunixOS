# Part II — Managing Memory

At the end of Part I the kernel is running, but it has no idea what memory it has. The GRUB bootloader placed the kernel binary somewhere above the 1 MB mark and provided a memory map, but the kernel has not read that map yet. Every address the kernel uses — the kernel stack, the screen output buffer, the temporary variables in C — sits in memory that has never been formally claimed. If the kernel were to allocate a buffer for a new data structure, it would have to pick an address by hand and hope nothing else was already there. That is not a foundation; it is a time bomb.

Memory management is not one problem but three nested problems that must be solved in order. First, the kernel needs to know which physical addresses contain usable RAM and which are reserved for hardware, firmware tables, or memory-mapped devices — the BIOS (Basic Input/Output System, the firmware that ran before GRUB) provides an E820 memory map that lists these regions, but it must be read and interpreted before anything overwrites it. Second, the kernel needs a way to allocate and free individual 4 KB pages of physical memory so that it can hand them out to data structures, page tables, and eventually user processes — a bitmap spanning all of usable RAM serves this role compactly. Third, the kernel needs a virtual address space: x86 paging is a hardware mechanism that translates the addresses C code generates into the physical addresses the memory bus sees, and setting it up correctly is what gives each future process its own isolated view of memory.

Chapter 7 reads the Multiboot memory map and builds the initial picture of physical RAM: which regions are usable, where the kernel image sits, and what must be avoided. Chapter 8 builds the entire memory stack on top of that picture — the physical memory manager's bitmap, the two-level x86 page directory and page table structure, the identity map of the first 128 MB that lets kernel virtual addresses equal physical addresses, and the `kmalloc`/`kfree` heap allocator that the rest of the kernel uses for dynamic allocation.

By the end of Part II the kernel has complete and correct control over RAM. It knows what physical pages exist, can allocate and free them, and has a virtual address space in place. Every subsystem built from this point forward can ask for memory and receive it safely.
